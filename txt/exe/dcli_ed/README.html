Syntax#

    -signature                          Show signatures and exit; alias -sig

Description#

In order to operate, DCLI tool needs ready-made acr data tables and script files sitting on local git repository and describing all actions necessary for building and deploying containers.

Ddcli_ed acts as composer, which prepares these data.

Tool function are:

  • manipulating (create, delete, rename or move, clone) dcli entities (dctr, dhost, dctrhost);

  • maintaining integrity of the database - sync with git, purge unnecessary files and records;

  • synchronizing with dcli - apply new config, deploy created (or cease deleted) containers

By default, instead of applying the changes immediately, dcli_ed produces an executable script on stdout. With -write, dcli_ed executes this script, which yields the same effect as piping output to sh.

For detailed description of dcli data model and related files, see dcli manual page.

Specifying entity#

Manipulated entity may be specified with corresponding option, like:

  -dctr app
  -dhost s1.example.com
  -dctrhost app.s1.example.com

Let us call this canonical form.

Alternatively, entitity may be specified with anonymous -target option, dcli_ed will guess target type.

Examples are:

  app                # target type is dctr (no dot)
  .s1.example.com    # target type is dhost (starts from dot)
  app.s1.example.com # target type is dctrhost

Note that last form is ambiguous, as it may be either dctrhost or dhost specified without preceding dot. Dcli_ed does its best to recognize, i.e. use context and checks for existence.

Let us call it universal form.

-cloneto and -moveto commands require argument in universal form.

Unless stated otherwise, both forms have identical effect. In further sections, we prefer canonical form.

Note that SQL regular expressions are supported for both forms (except -cloneto and -moveto where exact name is expected).

Specifying docker directory#

By default all scripts are stored on directory docker, to override this setting (e.g. for tests) use -script_dir.

Creating new dctr#

To create new dctr, use following syntax:

dcli_ed -create -dctr:X [-dctr_from:I] [-comment ...]

Override default starting image (ubuntu) with -dctr_from, put comment with -comment.

The command creates minimal workable container.

Creating new dhost#

To create new dhost, use following syntax:

dcli_ed -create -dhost:X -dhost_ip:IP -dhost_ext_ip:IP \
 [-dhost_ext_port:PORT] [-dhost_user:U] [-comment ...]

Specify host IP addresses with -dhost_ip and dhost_ext_ip mandatory options. Override default SSH port and user name with -dhost_ext_port and -dhost_user. Put comment with -comment.

As result, new dhost record is created, ssh config is updated.

Creating new dctrhost#

To create new dctrhost, use following syntax:

dcli_ed -create -dctrhost:X [-comment ...] [-force] [-sync:N] ...

Put comment with -comment.

Note that dctrhost requires corresponding dhost and dctr records. It is typical to create dctr along with dctrhost, so dcli_ed does it automaically. It is untypical that dhost does not exist, most probably it is a typo, dcli_ed denies creation of dctrhost. Use -force to create dhost automatically. For dhost and dctr creation, dcli_ed accepts corresponding options described in above sections.

Newly created dctrhost is immediately built and run on dhost. To skip this step, use ‘-sync:N’.

Deleting dctrhost#

To delete dctrhost, use following syntax:

dcli_ed -del -dctrhost:X [-sync:N]

As result, corresponding dctrhost record is deleted. Before making changes, dcli_ed stops and removes the container from docker host. To skip this step, put -sync:N. This is not recommended, as after making changes the container is no longer managed with dcli, and any actions must be done manually with docker commands.

When last dctrhost referencing some dctr is deleted, dctr is not deleted automatically, but becomes a subject to purge. One may invoke -purge command in the same shot to wipe unnecessary dctr.

Deleting dctr#

To delete existing dctr, use following syntax:

dcli_ed -del -dctr:X [-force] [-keep_files] ...

Dctr must not be referenced by any dctrhost, otherwise dcli_ed denies deletion. To delete all dctrhost using this dctr automatically, put ‘-force’, in that case dcthost deletion options apply, refer to above section.

As result, dctr record is removed from a database, along with its no longer necessary files on disk.

To keep scriptlets (.build, .run, .start), use -keep_files, note that it does not prevent from deletion of generated file.

Deleting dhost#

To delete existing dhost, use following syntax:

dcli_ed -del -dhost:X [-force] ...

Dctr must not be referenced by any dctrhost, otherwise dcli_ed denies deletion. To delete all dctrhost using this dctr, put ‘-force’, in that case dcthost deletion options apply, refer to above section.

Purging dctr and scripts#

To purge no longer necessary dctr (i.e. not referenced by any dctrhost), and all existing but not referenced files on script dir, use:

dcli_ed -purge

Note that -keep_files has no effect for this command.

Fundamental cases are: dctr-to-dctr, dhost-to-dhost. Other cases are just for convenience, they aggregate several fundamental actions.

Note that cloning function never overwrites existing entities or script files.

Cloning dctr#

To clone existing dctr X to new dctr Y, use the following syntax:

dcli_ed -dctr:X -cloneto:Y [-deep]

By default, scriptlet files (.build, .run, .start) are reused for new dctr. To make own copy of them, use -deep.

As result, a copy of dctr with all child dctrua records, and optionally script files is made.

Cloning dhost#

To clone existing dhost X to new dhost Y along with with all child containers, use the following syntax:

dcli_ed -dhost:X -cloneto:Y

As result, source dctrhost cloned with new name, along with all child dctrhost records.

Rename or move#

Renaming or moving semantics is straightforward:

Source

Target

Behavior

dctr

dctr

rename dctr

dhost

dhost

rename dhost

dctrhost

dhost

move dctrhost to other existing dhost

dctrhost

dctrhost

move dctrhost to other existing dhost (dctr must be the same)

Other cases make no sense.

Rename dctr#

To rename existing dctr X to new name Y, use the following syntax:

dcli_ed -dctr:X -renameto:Y [-keep_files] [-deep]

By default, scriptlets are renamed in smart manner: all occurrences of old dctr name are replaced by new one. For example: old container name: my-app, new container name my-app-new, script file docker/app/my-app.start will be renamed to docker/app/my-app-new.start.

To force renaming to default form docker_dir/dctr/dctr.{build,run,start} whatever the source name, use -deep For the example above, script file will be renamed to docker/my-app-new/my-app-new.start.

To suppress renaming use -keep_files, note this option takes precedence over -deep.

As result, dctr is renamed, all child containers are updated. All dctrs reusing renamed scripts will also be updated.

Rename dhost#

To rename existing dhost X to new name Y, use the following syntax:

dcli_ed -dhost:X -renameto:Y

Move dctr from one dhost to another#

To rename existing dctrhost X to new name Y, use use the follwging syntax:

dcli_ed -dctrhost:X -renameto:Y

Note that dctr part of new name shall be identical to old one, or just omitted.

Examples#

  • create docker host dcli_ed -create .s1.example.com -dhost_ip:192.168.0.10 -dhost_ext_ip:dmz.example.com

  • create and run fresh minimal container dcli_ed -create app.s1.example.com

  • delete container and clean up after it dcli_ed -del app.s1.example.com -purge

  • create clone of existing container (on the same or different host) dcli_ed app.s1.example.com -cloneto app-clone.s1.example.com dcli_ed app.s1.example.com -cloneto app.s2.example.com

  • rename container dcli_ed app -renameto newapp

  • rename host dcli_ed s1.example.com -renameto snew.example.com

  • move container from one host to another dcli_ed app.s1.example.com -renameto app.s2.example.com
